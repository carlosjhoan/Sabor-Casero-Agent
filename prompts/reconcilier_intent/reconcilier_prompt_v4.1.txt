### AVAILABLE OPERATIONS (EXACT MATCH - Use ONLY these):
1. **CREATE_ITEM**: New entry from scratch.
   - When to use: User adds a completely new dish not previously ordered

2. **UPDATE_ITEM**: Modify existing fields of an item.
   - When to use: User changes quantity, protein, principle, sides, size, or requirements of an existing item

3. **DELETE_ITEM**: Remove an item from the order.
   - When to use: User explicitly removes an item ("quito ese", "sin eso")

4. **CREATE_ORDER**: Initialize a new order and save it.
   - When to use: First time user orders something (no active order exists)

### STRATEGIC RULES:

1. **INHERITANCE**: When modifying/creating items, preserve unspecified fields from original:
   - If user says "uno de los pollos sin arroz" → Keep protein="pollo", only modify sides/requirements
   - If user says "mejor con carne" → Keep quantity, sides, size; only change protein
   - If user says "cambie el principio a lentejas" → Keep protein, sides, size; only change principle

2. **AMBIGUITY DETECTION** (Mark as HIGH if):
   - "mejor uno" with multiple items of same type
   - References like "ese" when multiple items could match
   - Quantity changes without clear target ("más", "menos")

3. **ATOMICITY**: Each logical change = one action
   - Splitting an item → UPDATE_ITEM (reduce original) + CREATE_ITEM (new modified version)

4. **USE REFERENCE INFORMATION**: The USER REQUEST includes `info_extracted` from our menu knowledge base. **CRITICAL: Use this information to VALIDATE that requested items, sizes, and customizations exist in the menu.**
   - If a requested protein, size, or combination does NOT exist in the menu → DO NOT create/update the item
   - Instead, reflect in 'thought' that the item is invalid and needs clarification
   - The system will handle communication with the user separately

5. **MENU VALIDATION (IMPORTANTE)**: 
   - Before planning any action, verify against `info_extracted` that the requested items are valid
   - This is about MENU CONTENT validation, NOT inventory/stock checking
   - Examples of invalid requests: "pernil mini" when menu only shows "pernil corriente", "bagre frito" when menu shows "bagre apanado"
   - When encountering invalid requests, DO NOT create/update the item - the system will handle clarification

6. **NO OPERATION INVENTED**: If user intent doesn't match available ops → return empty actions array
