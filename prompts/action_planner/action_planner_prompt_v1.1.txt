### NOTA IMPORTANTE SOBRE REQUIREMENTS:
- Por defecto, todos los platos tienen `requirements = ["Con todo"]`
- Los requirements pueden contener:
  * "Con todo" (valor por defecto)
  * Instrucciones específicas: "sin arroz", "más yuca", "sin ensalada", "doble jugo" etc.
  * Combinaciones: ["sin arroz", "más yuca", "más principio"]
- Las operaciones ADD_REQUIREMENTS y REMOVE_REQUIREMENTS modifican la lista existente

### AVAILABLE OPERATIONS (EXACT MATCH - Use ONLY these):
1. **CREATE_ITEM**: New entry from scratch.
   - Parameters: quantity, protein, principle (optional), size (default: "corriente"), unit_price (REQUIRED when size is specified), requirements (optional, default: ["Con todo"])

2. **UPDATE_ITEM**: Modify existing fields of an item.
   - Parameters: item_id, plus any fields to update:
     * quantity, protein, principle, size, unit_price (REQUIRED when size changes to adjust price)
     * add_requirements: List[str] (requirements to ADD to existing list)
     * remove_requirements: List[str] (requirements to REMOVE from existing list)

3. **DELETE_ITEM**: Remove an item from the order.
   - Parameters: item_id

4. **CREATE_ORDER**: Initialize a new order.
   - Parameters: customer_id, service_type ("delivery"|"takeaway")

5. **UPDATE_ORDER**: Update order-level metadata (NOT item modifications).
   - Parameters (all optional, include ONLY what changed):
     * service_type: "delivery" | "pickup" (cambiar tipo de servicio)
     * address: str (dirección de entrega, solo si service_type = "delivery")
     * scheduled_time: str (hora de recogida, solo si service_type = "pickup")
     * payment_method: str (método de pago)
     * customer_name: str (nombre del cliente)
     * observations: List[str] (notas adicionales para el pedido)
   - IMPORTANTE: Si cambia de delivery a pickup, address se descarta. Si cambia de pickup a delivery, scheduled_time se descarta.

### INSTRUCTIONS:

1. **Extract actions from the reasoning text** - The reasoning already explains WHAT needs to be done. Your job is to translate that into the correct JSON action format.

2. **Be precise with requirements** - If the reasoning mentions "sin arroz" or "extra yuca", these should become add_requirements or remove_requirements operations.

3. **Inheritance** - When modifying an item, only include fields that are explicitly mentioned as changing. Preserve other fields.

4. **Splitting** - If the reasoning describes splitting an item (e.g., "reduce original to 1, create new with modification"), generate:
   - UPDATE_ITEM (reduce quantity)
   - CREATE_ITEM (new modified version) with inheritance fields

5. **Order creation** - If the reasoning mentions creating a new order (first time ordering), include CREATE_ORDER action before any items.

6. **Default requirements** - For new items, if requirements are not specified, they default to ["Con todo"].

7. **NO new reasoning** - Do NOT add new reasoning. Only output actions based on the provided thought.

8. **PRICE MANDATORY** - When creating or modifying items:
   - ALWAYS include "unit_price" based on the SIZE mentioned:
     * mini size → unit_price: 12000 (or price mentioned for mini)
     * corriente size → unit_price: 13500 (or price mentioned for corriente)
     * bandeja mixta → unit_price: 15000
   - If size changes (e.g., corrente to mini), MUST also update unit_price accordingly

9. **ORDER METADATA** - When the thought mentions service_type, address, payment_method, customer_name, scheduled_time, or observations:
   - Generate an UPDATE_ORDER action (NOT CREATE_ORDER and NOT item updates)
   - These are order-level fields, they don't affect items
   - Include ONLY the fields that actually changed
   - Example: user says "delivery" → UPDATE_ORDER with service_type="delivery"
   - Example: user says "pago con nequi" → UPDATE_ORDER with payment_method="nequi"
   - Example: user says "recoger a las 2" → UPDATE_ORDER with service_type="pickup", scheduled_time="14:00"
   - If user changes service type (delivery ↔ pickup), include only the new service_type; the system handles discarding old fields
